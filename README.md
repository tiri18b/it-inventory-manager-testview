# IT Inventory Manager

כלי Desktop פשוט לניהול ציוד מחשוב בחברה.

המערכת נבנתה במסגרת משימת בית עבור Testview, במטרה להציג חשיבה מסודרת, יכולת למידה, ממשק משתמש ברור ופתרון עובד לניהול מלאי IT בסיסי.

## טכנולוגיה

C#
.NET 8
Windows Forms
שמירת נתונים מקומית בקובץ JSON

## מה מומש במערכת

הוספת פריטי ציוד חדשים
עריכת פריטי ציוד קיימים
מחיקת פריטים
חיפוש חופשי לפי תג ציוד, עובד, דגם, יצרן, מספר סידורי ועוד
סינון לפי סטטוס
סינון לפי סוג ציוד
שיוך ציוד לעובד
הצגת סטטוס ציוד: In Stock, In Use, Faulty
תצוגת סיכום של כמות ציוד כללית, ציוד בשימוש, ציוד במלאי וציוד תקול
שמירת הנתונים לקובץ JSON מקומי
טעינת נתוני דוגמה בהרצה ראשונה

## איך מריצים את המערכת

נדרש מחשב Windows עם .NET 8 SDK או Visual Studio 2022.

אפשרות 1: הרצה דרך Visual Studio

1. לפתוח את הקובץ `ITInventoryManager.csproj`
2. ללחוץ Build
3. ללחוץ Start

אפשרות 2: הרצה דרך PowerShell מתוך תיקיית הפרויקט

```powershell
cd "<PROJECT_FOLDER>"
dotnet restore .\ITInventoryManager.csproj
dotnet build .\ITInventoryManager.csproj
dotnet run --project .\ITInventoryManager.csproj
```

## שמירת נתונים

המערכת שומרת את הנתונים בקובץ JSON מקומי שנוצר אוטומטית בזמן הריצה.

הקובץ נוצר תחת תיקיית ההרצה של האפליקציה:

```text
bin\Debug\net8.0-windows\Data\inventory-data.json
```

אם הקובץ לא קיים, המערכת יוצרת כמה פריטי דוגמה כדי שיהיה קל לבדוק ולהציג את המערכת.

## יצירת קובץ EXE להגשה או להרצה

אפשר ליצור גרסת Release עם קובץ EXE מתוך PowerShell בתיקיית הפרויקט.

```powershell
cd "C:\Users\tiri1\Desktop\Sites\test view\ITInventoryManager\testview_it_inventory"

.\publish-exe.ps1
```

הקבצים ייווצרו בתיקייה:

```text
publish\win-x64
```

הקובץ המרכזי להרצה יהיה:

```text
ITInventoryManager.exe
```

הפקודה יוצרת גרסת Windows x64 במצב Release. בחרתי להכין אפשרות פרסום כ־self-contained כדי שיהיה קל יותר להריץ את האפליקציה גם במחשב שאין עליו סביבת פיתוח מלאה.

## מה הייתי מוסיף אם היה יותר זמן

בשלב הראשון בחרתי לבנות מערכת פשוטה וברורה כדי לעמוד בזמן המשימה ולהציג פתרון עובד.

אם היה לי זמן נוסף, הייתי ממשיך לפתח את הכלי הזה לכיוון של מערכת פנימית אמיתית שמייעלת את העבודה של צוות IT ושל עובדי החברה.

### התחברות משתמשים והרשאות

משתמש רגיל יוכל לצפות בציוד או לפתוח בקשה, ואיש IT או מנהל יוכל להוסיף, לערוך ולמחוק פריטים. זה חשוב כדי למנוע שינויים לא מורשים ולשמור על אמינות הנתונים.

### היסטוריית שינויים ובקרה

מעקב אחרי מי שינה כל פריט, מתי ציוד שויך לעובד, מתי הוא הוחזר למלאי ומתי סומן כתקול. זה חשוב לאחריות, בקרה, סדר בעבודה וחזרה אחורה במקרה של טעות.

### ייבוא וייצוא לאקסל

הרבה חברות כבר מנהלות רשימות ציוד באקסל. ייבוא וייצוא היו מקלים על מעבר למערכת, מאפשרים הפקת דוחות מהירה ושיתוף מידע עם הנהלה או כספים.

### תמיכה בברקוד ו־QR Code

כל פריט ציוד יקבל ברקוד או QR Code. איש IT יוכל לסרוק מחשב, מסך או ציוד אחר, לפתוח מיד את הרשומה שלו ולעדכן סטטוס, מיקום או שיוך לעובד. זה יכול לחסוך זמן בספירות מלאי ובטיפול בתקלות.

### יצירת מדבקות QR להדפסה

המערכת תוכל ליצור מדבקות QR מוכנות להדפסה עבור ציוד חדש, כולל תג ציוד, סוג ציוד וקוד סריקה. כך בזמן קליטת ציוד חדש אפשר להדפיס מדבקה, להדביק על הפריט, ולנהל את כל החיים של הציוד בצורה מסודרת.

### מעבר ל־SQL Server וניהול נתונים נכון

בגרסה ארגונית הייתי מעביר את הנתונים מקובץ JSON למסד נתונים מרכזי כמו SQL Server. זה יאפשר עבודה של כמה משתמשים במקביל, שמירה אמינה יותר, הרשאות, שאילתות, דוחות, וגיבוי מסודר של המידע.

### גיבוי מקומי וגיבוי בענן

הייתי מוסיף מנגנון גיבוי אוטומטי: גיבוי מקומי לשרת פנימי וגיבוי מוצפן לענן. כך אם מחשב נופל, קובץ נמחק או יש תקלה, אפשר לשחזר את המידע במהירות. בחברה אמיתית, מערכת מלאי בלי גיבוי היא סיכון תפעולי.

### ניהול עובדים ומחלקות

אפשרות לראות איזה ציוד נמצא אצל כל עובד, איזה ציוד שייך לכל מחלקה, ומה צריך להחזיר בזמן עזיבת עובד. זה יכול לעזור מאוד בתהליכי קליטת עובד חדש וסיום עבודה של עובד.

### תהליך קליטת עובד ועזיבת עובד

הייתי מוסיף תהליך מובנה ל־Onboarding ו־Offboarding. לדוגמה, בעת קליטת עובד חדש המערכת תציג רשימת ציוד שצריך להכין עבורו, ובעת עזיבה היא תציג רשימת ציוד שצריך להחזיר. זה חוסך טעויות ומייעל עבודה מול משאבי אנוש ומנהלים.

### מעקב אחר אחריות וספקים

הוספת תאריך רכישה, תוקף אחריות ופרטי ספק. זה יכול לחסוך כסף לחברה, כי לפני שמחליפים ציוד אפשר לבדוק אם הוא עדיין באחריות ומול מי צריך לפתוח קריאת שירות.

### תהליך טיפול בציוד תקול

במקום רק לסמן ציוד כתקול, הייתי מוסיף תהליך טיפול: תיאור התקלה, מי מטפל, תאריך פתיחה, סטטוס תיקון, הערות טכנאי ותאריך חזרה למלאי. כך ציוד תקול לא נעלם מהמעקב.

### דשבורד ודוחות להנהלה ול־IT

דוחות על ציוד במלאי, ציוד בשימוש, ציוד תקול, ציוד לפי עובד, ציוד לפי מחלקה, ציוד ישן שדורש החלפה, ועלויות ציוד לפי תקופה. דשבורד כזה יכול לעזור ל־IT ולהנהלה לקבל החלטות מבוססות נתונים.

### התראות ותזכורות

התראות על אחריות שעומדת להסתיים, ציוד תקול שלא טופל הרבה זמן, ציוד שלא שויך לעובד, או ציוד ישן שצריך להחליף. זה הופך את העבודה מפרואקטיבית במקום תגובתית.

### אינטגרציה למערכות ארגוניות

בגרסה מתקדמת ניתן לחבר את המערכת למערכות קיימות כמו Priority או מערכת קריאות שירות, כדי לקשר בין ציוד, עובדים, רכישות, תקלות ותהליכים פנימיים. זה יכול להפוך את המערכת מכלי מלאי פשוט לכלי שמייעל תהליכים אמיתיים בחברה.

### שימוש ב־AI לייעול עבודה

אפשר לשלב AI כדי לסכם תקלות, להציע פעולות בדיקה לפי סוג ציוד, לנסח תיעוד טכני, ולזהות ציוד שחוזר לתקלות חוזרות. המטרה היא לא להחליף איש IT, אלא לעזור לו לעבוד מהר ומסודר יותר.
